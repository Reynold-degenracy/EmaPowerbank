# 反馈功能说明

## 功能入口

普通用户登录后，侧边栏会显示与“面板”“日志”同级的“反馈”入口。管理员端不显示普通反馈入口，只显示独立的“反馈审核”入口。

反馈表单包含：

- 必填文字描述
- 可选图片附件

反馈通过管理员审核后，会获得一定美元余额奖励。奖励金额由管理员审核时选择，未审核或被拒绝的反馈不会发放奖励。

## 提交接口

前端通过 `POST /api/feedback` 提交反馈。接口要求用户已登录，并使用 `multipart/form-data`。

表单字段：

- `description`：必填，反馈文字描述，提交前后端都会去除首尾空白，最大 5000 字符
- `attachment`：可选，图片附件

附件支持常见图片格式：

- PNG
- JPEG / JPG
- GIF
- WebP
- BMP
- AVIF
- HEIC / HEIF

单个附件最大 8 MB。前端会先做格式和大小校验，后端仍会再次校验。

## 保存位置

后端会在项目运行目录下创建 `feedback/`，每次提交保存为一个独立目录：

```text
feedback/feedback-<id>-<timestamp>/
```

示例：

```text
feedback/feedback-fb_3d75a673-5aaf-4c60-9342-472feac83378-2026-06-09T13-38-03-368Z/
```

其中：

- `feedback-` 是固定前缀
- `fb_3d75a673-5aaf-4c60-9342-472feac83378` 是反馈 id，从 `fb_` 开始
- `2026-06-09T13-38-03-368Z` 是提交时间戳

目录名中的时间戳会把 ISO 时间里的 `:` 和 `.` 替换成 `-`，方便作为文件系统路径使用。原始时间戳仍保存在 `feedback.json` 的 `timestamp` 字段里。

## 反馈包内容

每个反馈目录固定包含：

```text
feedback.json
```

如果用户上传了图片，还会包含一个图片文件，文件名同样包含反馈 id 和时间戳：

```text
feedback-fb_<uuid>-<timestamp>.<ext>
```

`feedback.json` 包含：

- `id`：反馈 id
- `timestamp`：原始 ISO 提交时间
- `packageName`：反馈包目录名
- `user`：提交用户的 id、用户名和角色
- `description`：反馈文字描述
- `attachment`：附件元数据；没有附件时为 `null`
- `review`：审核状态和奖励信息

附件元数据包含：

- `fileName`：后端保存后的附件文件名
- `originalName`：用户上传时的原始文件名
- `mimeType`：附件 MIME 类型
- `size`：附件字节数

`review` 包含：

- `status`：`pending`、`approved` 或 `rejected`
- `rewardAmount`：审核通过时发放的奖励点数，待审核或拒绝时为 `0`
- `reviewedBy`：审核管理员的 id、用户名和角色；未审核时为 `null`
- `reviewedAt`：审核时间；未审核时为 `null`

旧反馈文件如果没有 `review` 字段，会按 `pending` 处理。

## 管理员审核

管理员可以通过侧边栏独立的“反馈审核”入口打开审核面板。

审核面板支持：

- 按 `待审核`、`已通过`、`已拒绝` 筛选反馈
- 查看反馈提交人、提交时间、描述、包名和附件信息
- 对图片附件进行预览
- 拒绝反馈
- 通过反馈并发放奖励点数

通过反馈时，奖励金额必须大于 `0`。界面提供常用档位 `$1.00`、`$3.00`、`$5.00`、`$10.00`，同时保留手动填写能力，手动输入步进为 `$0.10`。

一个反馈只能审核一次。已通过或已拒绝的反馈不能再次通过，也不会重复发放奖励。

审核信息写回该反馈包里的 `feedback.json`。奖励点数会增加到提交用户现有的账户余额中。

## 运行与维护

`feedback/` 是运行时数据目录，已经加入 `.gitignore`，不会进入版本库。

生产环境建议：

- 限制 `feedback/` 的文件权限
- 按需清理、归档或备份反馈包
- 不要把包含用户反馈或附件的目录提交到代码仓库
